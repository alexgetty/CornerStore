Design Token Naming Conventions

Research report for Corner Store Storefront theming system
March 2026 · 11 systems surveyed, ~40 sources consulted

Scope. This report studies how open-source design systems name their CSS custom property tokens -- specifically the naming patterns that end users interact with. It does not cover token architecture, build tooling, or internal-only systems. Every claim is sourced; gaps are noted explicitly.

2. Convergence Summary

Across the 11 systems studied, certain patterns recur frequently enough to be considered de facto standards. Others remain genuinely contested. Here is what I found.

2.1 Where Systems Converge Strong Agreement

Prefix for collision avoidance

Every system that ships as a library uses a prefix: --sl-, --pico-, --chakra-, --md-, --slds-, --spectrum-, --ds-, --n-, --cds-. The only exceptions are shadcn/ui and Radix Themes, which generate project-local CSS and don't need to avoid collisions with other libraries. For any system that ships as a package, a short prefix is universal practice.

Semantic (role-based) over functional (property-based) for theme tokens

Every system studied uses semantic naming for its theme-level color tokens. No system asks users to set --background-color-of-card. They all abstract to role: primary, surface, accent, neutral, muted, destructive/error/danger. The functional/property layer lives underneath (or doesn't exist at all).

Background/foreground pairing with consistent suffix

Every system that exposes user-facing color tokens pairs backgrounds with foreground/text tokens. The suffix varies, but the pattern is universal:

SystemBackgroundForeground suffix
shadcn/ui--primary--primary-foreground
DaisyUI v5--color-primary--color-primary-content
Material Design 3--md-sys-color-primary--md-sys-color-on-primary
Chakra UIcolors.bgcolors.fg
Nord Health--n-color-surface--n-color-text
Open Props--surface-{n}--text-{n}
Radix Themes--color-background(implicit via scale steps)
Atlassian--ds-background-*--ds-text-*

No abbreviations in public token names

The clear trend is toward full words. Pico spells out background-color. Shoelace spells out border-radius. Material spells out everything including tier markers. Nord explicitly recommends full words in their naming docs. DaisyUI's migration FROM abbreviations (--p, --b1) TO full words (--color-primary, --color-base-100) is the strongest evidence that abbreviations are a mistake for public APIs.

The only common exception: bg/fg appear in Chakra UI and Tailwind utility classes, where brevity in heavily-repeated contexts is valued. But even Chakra uses full words for its CSS custom properties (--chakra-colors-bg is a semantic token name, not an abbreviation -- "bg" is the concept).

Kebab-case with hyphens

Universal. Every system uses kebab-case for CSS custom properties. No camelCase, no underscores (outside of Open Props' briefly-considered --_ prefix).

Foreground naming: -foreground vs. -content vs. on- vs. -text

No single winner. -foreground (shadcn), -content (DaisyUI), on- prefix (Material), and separate text category (Open Props, Atlassian, Nord) are all in active use.

Color hierarchy: numbered vs. named emphasis

Open Props uses --text-1, --text-2 (numbered emphasis). shadcn uses named roles (--muted, --accent). Atlassian uses adjective intensity (-subtle, -bold). Material uses primary/secondary/tertiary. No convergence.

2.3 Patterns with Emerging Consensus Trending

"surface" has become standard terminology

Used by: Material Design 3 (surface, surface-container, on-surface), Open Props (--surface-1 through --surface-4), Radix Themes (--color-surface, --accent-surface), Nord Health (surface role), Atlassian (--ds-surface-raised, --ds-surface-overlay), SLDS (--slds-g-color-surface-container-1).

Not used by: shadcn/ui (uses --background), Pico CSS (uses --background-color), DaisyUI (uses --color-base-100).

"Surface" is not CSS-native vocabulary, but it has been adopted by enough major systems that it is no longer jargon -- it is emerging standard terminology for "background of a container/component."

"primary/secondary/accent" hierarchy is dominant

Nearly every system uses primary. Most add secondary. Several include accent. tertiary appears mainly in Material Design. The broader community has largely settled on primary + secondary + accent as the standard three-level brand color hierarchy.

State colors: success, warning, error/danger/destructive

success and warning are universal. The "bad" state varies: error (DaisyUI, Material, Carbon), danger (Shoelace), destructive (shadcn). info appears in DaisyUI, Shoelace, and Chakra but not universally.

3. Specific Questions Answered

3.1 Is "surface" jargon or standard?

It has become standard. Six of the eleven systems surveyed use "surface" as a token name for background colors. Material Design -- which has enormous reach through Android and Flutter -- made it a first-class concept. Open Props, Radix, Nord, Atlassian, and SLDS followed. It still requires a one-sentence explanation for someone encountering it for the first time, but that sentence is "surface = background of a thing." The Figma design community also uses "surface" as a standard element role for color variables.

Verified via: Material color roles, Open Props, Radix color docs, Nord naming, Atlassian tokens

3.2 "foreground/background" or "surface/text"?

These are two different framing decisions that both work. The key distinction:

  • "foreground/background" treats colors as a pair on the same concept. shadcn: --primary (bg) + --primary-foreground (text). DaisyUI: --color-primary + --color-primary-content. The pair belongs together.
  • "surface/text" treats colors as separate categories. Open Props: --surface-1 + --text-1. Atlassian: --ds-background-* + --ds-text-*. The categories are independent; you combine them yourself.

For a small token set (~45 tokens) aimed at non-expert users, the separate-category approach (surface + text) is simpler to explain: "here are your backgrounds, here are your text colors." The paired approach requires understanding that --primary and --primary-foreground are a unit.

3.3 Primary/secondary/tertiary hierarchy

primary is universal. secondary is very common. tertiary is Material-specific and rarely adopted elsewhere. Most systems use accent instead of (or alongside) secondary. DaisyUI uses all three: primary, secondary, accent. shadcn uses primary, secondary, accent, muted.

Multiple authors note that beyond "tertiary" the Latin ordinals (quaternary, quinary) are impractical. Systems that need more levels switch to descriptive names or numeric scales.

Verified via: DaisyUI colors, shadcn theming, Material color roles

3.4 Spell out "background" or abbreviate to "bg"?

The weight of evidence favors spelling it out:

  • Pico: --pico-background-color, --pico-card-background-color (full)
  • SLDS: --slds-g-color-background-* (full)
  • Atlassian: --ds-background-* (full)
  • Chakra: --chakra-colors-bg (abbreviated, but "bg" is the semantic concept name)
  • Tailwind: bg- in utility classes, but --color- in theme variables (no "bg" in custom properties)

Most systems that expose CSS custom properties spell out "background." Abbreviation bg is more common in utility-class frameworks (Tailwind) where it's typed repeatedly. For token names that appear once in a theme file, the consensus favors readability.

However: Corner Store uses --cs-interactive-bg in its current draft. This is defensible if "interactive" is the concept and "bg" is a well-understood shorthand for a property within that concept. But it mixes an abbreviated property (bg) with spelled-out concepts, which creates inconsistency with tokens like --cs-surface-page and --cs-border-color.

3.5 Interactive element tokens (buttons, links)

Approaches vary widely:

  • Material: No dedicated interactive tokens. Buttons use primary, secondary, tertiary, plus component tokens (--md-filled-button-container-color).
  • Pico: Interactive states on the color role: --pico-primary-hover, --pico-primary-focus.
  • Carbon v10: Dedicated interactive concept: interactive-01, interactive-02.
  • Shoelace: Component tokens: --sl-button-font-size-medium, --sl-input-background-color-hover.
  • shadcn: No interactive-specific tokens. Components consume --primary and --primary-foreground.

The trend is away from dedicated "interactive" tokens and toward components consuming the semantic color roles directly. "Interactive" as a token category is Carbon-era thinking; most modern systems let components reference primary.

The problem with comparatives is twofold: (1) they imply a complete ordering that prevents inserting new sizes, and (2) smallest suggests "the smallest possible" rather than "a small size within the system."

3.8 Is there a standard token name grammar?

No single standard, but a clear pattern emerges across expert guidance and implementations:

--{prefix}-{category}-{concept}-{property}-{modifier}

Where:
  prefix   = namespace (cs, sl, pico, md)
  category = color, font, space, radius, shadow, border
  concept  = surface, text, interactive, heading, button
  property = size, weight, color, width (often implied)
  modifier = variant (hover, muted, large) or scale (1, 2, sm, lg)

Not every token uses every level. Most small systems use 2-3 levels. SLDS uses up to 7. The key insight from Nathan Curtis (EightShapes): "Namespaces first, base levels in the middle, modifiers last."

Source: Nathan Curtis, "Naming Tokens in Design Systems", Smashing/Vitaly Friedman article

4. Renaming History and Lessons

4.1 DaisyUI v4 to v5: Abbreviations to Full Words

DaisyUI v4 used aggressively abbreviated CSS variable names: --p (primary), --s (secondary), --b1 (base-100), --pc (primary-content), --sf (secondary-focus). v5 renamed every token to readable full words aligned with Tailwind v4's naming convention: --color-primary, --color-base-100, --color-primary-content.

This was a full breaking change requiring manual migration. The rename was motivated by Tailwind v4's move to CSS-first configuration with semantic variable names. The lesson: abbreviated names that save keystrokes in authoring become opaque API contracts that resist documentation and onboarding.

Source: DaisyUI v5 upgrade guide, v5 upcoming changes

4.2 IBM Carbon v10 to v11: Numbered to Descriptive

Carbon v10 used numeric suffixes: interactive-01, text-01, text-02, ui-01, ui-02, ui-03. Users couldn't tell what the numbers meant without consulting documentation. v11 replaced numbers with descriptive adjectives and renamed categories: text-primary, text-secondary, layer-01. Only layering tokens retained numeric suffixes (where the number has genuine sequential meaning).

The rename affected 52 universal color tokens per theme. The new grammar is: [element]-[role]-[order]-[state].

Source: Carbon migration guide

4.3 Open Props: Considered and Rejected Prefix Change

In Issue #160, Adam Argyle considered adding an underscore prefix to theme tokens (--_surface-1 instead of --surface-1) to distinguish them from generic props. He closed the issue after deciding the distinction was unnecessary, calling it "trivial and silly." The names shipped unprefixed. This shows that even framework authors agonize over naming, and simplicity often wins over clever disambiguation.

4.4 Atlassian Design Tokens: Structured Deprecation

Atlassian documented a formal deprecation lifecycle: tokens marked deprecated in a minor version, soft-deleted (functional but raising warnings) in the following minor, fully removed in the next major. This three-phase approach (deprecate, soft-delete, remove) is the most structured migration process found in any system surveyed.

Source: Atlassian developer docs

4.5 Lessons Synthesized

  • Abbreviations are technical debt. Both DaisyUI and Carbon renamed away from abbreviated/numbered tokens toward descriptive names. Neither rename was trivial.
  • Names that need documentation are bad names. If a user must look up what --b1 or --text-02 means, the name has failed its primary job.
  • Numbers work when they have inherent meaning. Radix's --space-1 through --space-9 works because the numbers form a sequence. Carbon's text-01 failed because "01" doesn't tell you what kind of text.
  • Plan for deprecation. Alias old names to new ones during migration. Atlassian's three-phase approach is the gold standard.

5. Expert Naming Guidance

5.1 Nathan Curtis (EightShapes)

Curtis's taxonomy framework, documented in "Naming Tokens in Design Systems" (2020), is the most-cited reference in this space. Key points:

  • Token names combine up to six levels: namespace, category, concept, property, variant, scale.
  • Example: $esds-color-neutral-42 = namespace (esds) + category (color) + variant (neutral) + scale (42).
  • "Namespaces first, base levels in the middle, modifiers last."
  • Mode modifiers (light/dark) can be appended but are often handled at the declaration level rather than in the name.

Note: The Medium article returned 403 when fetched directly. Key claims above are from search result excerpts and consistent with Curtis's other published work.

5.2 Vitaly Friedman (Smashing Magazine / Smart Interface Design Patterns)

The naming article synthesizes patterns across systems:

  • Naming levels: category, component/concept, property, variant/scale, state, mode.
  • "Regular systems typically use 2-4 levels."
  • "Good tokens explain what they do."
  • Namespaces first, base levels in middle, modifiers last -- consistent with Curtis.

5.3 W3C Design Tokens Community Group

The DTCG specification (2025.10 stable) intentionally does not prescribe naming conventions. It specifies:

  • Names must be valid JSON strings, are case-sensitive.
  • Characters $, {, }, . are restricted.
  • "Groups are arbitrary and tools should not use them to infer the type or purpose of design tokens."
  • Hierarchical paths use . as separator (e.g. color.brand.primary).

The spec stays naming-agnostic by design, leaving conventions to individual systems.

5.4 Smashing Magazine (2024)

The "Best Practices for Naming" article collects guidance:

  • Use full words instead of abbreviations. Abbreviations are unclear and open to interpretation.
  • Be consistent with prefixes.
  • "A good name has a logical structure, is short, meaningful, and known by everyone."
  • References Romina Kavcic's Design Tokens Naming Playbook for structured naming workshops.

6. Recommendations for Corner Store

These recommendations apply the research findings to Corner Store's specific context: indie makers with basic CSS knowledge, ~45 tokens, --cs- prefix, semver-bound at v1.

Framing. Corner Store's users are not design system engineers. They are people who make candles and hot sauce. They know enough CSS to change a color value but should never need to decode a naming convention. Every token name should be self-evident on first encounter.

6.1 Keep the --cs- prefix

Status: Already correct. As an npm package, Corner Store needs a namespace prefix. --cs- is short (2 chars), memorable, and avoids collision. This aligns with universal practice across every packaged system surveyed.

6.2 Keep "surface" for backgrounds

Status: Already correct. --cs-surface-page, --cs-surface-listing, --cs-surface-header align with the emerging standard. Six of eleven systems surveyed use "surface." A one-line comment in the generated theme file ("surface = background color") handles onboarding.

The current names are also better than generic numbered --cs-surface-1 tokens because they name the THING (page, listing, header) rather than an abstract level. A candle maker knows what "listing" means. They don't know what "surface-2" means.

6.3 Keep "text" as a separate category from "surface"

Status: Already correct. --cs-text-primary, --cs-text-secondary, --cs-text-muted follow the Open Props / Atlassian / Nord pattern of separate text and surface categories. This is simpler to explain than the paired approach (--primary / --primary-foreground), which requires understanding that tokens come in pairs.

Corner Store doesn't need per-surface foreground tokens because the use case is simple: one page, one text hierarchy, applied across all surfaces. shadcn needs --card-foreground because a card might have different text than the page. A maker's storefront does not have this complexity.

6.4 Rename "interactive" to spelled-out names

Recommendation: Rename. The current --cs-interactive-bg, --cs-interactive-bg-hover, --cs-interactive-text have two problems:

  1. bg is an abbreviation in a system that otherwise spells things out (surface, border-color, font-family). This inconsistency was noted in the survey.
  2. "Interactive" is jargon that a candle maker will not immediately understand. The research shows the trend is away from dedicated "interactive" tokens and toward buttons/CTAs consuming brand-role tokens directly.

Two paths forward:

Option A: Rename to --cs-action-* with spelled-out properties.

--cs-action-background       /* Button/CTA background */
--cs-action-background-hover /* Button hover state */
--cs-action-text             /* Button/CTA text color */

"Action" is shorter than "interactive" and more intuitive -- a button IS an action. But it still introduces a concept name for what is essentially "the button color."

Option B: Rename to --cs-button-*.

--cs-button-background       /* Button/CTA background */
--cs-button-background-hover /* Button hover state */
--cs-button-text             /* Button/CTA text color */

Most direct. A maker sees "button" and knows exactly what it styles. The concern is that it's component-specific, which could feel limiting. But Corner Store has exactly one interactive element (the buy CTA). Naming it what it is has no downside.

I would lean toward Option B based on the research principle "name the thing, not the abstraction," which is already embedded in Corner Store's own selector naming philosophy (.cs-listing-buy, not .cs-interactive-element). But this is a one-way-ish decision that deserves your judgment.

6.5 Reconsider the font-size comparative scale

Recommendation: Rename. The current scale (smallest, smaller, small, base, large, larger, largest) has no precedent in any system surveyed. Two concerns from the research:

  1. Extensibility. "Smallest" and "largest" are superlatives that semantically close the scale. What if you need something smaller than "smallest" later? T-shirt sizes (xs, 2xs) or numbers can always extend.
  2. Familiarity. Every developer who has used Tailwind, Chakra, Shoelace, or Spectrum has internalized the t-shirt / multiplier pattern. Comparative adjectives are a convention of one.

Recommended alternatives (in order of preference for Corner Store's audience):

Option A: T-shirt with spelled-out words. Follows Shoelace's lead. Maximum readability for CSS-literate non-experts.

--cs-font-size-x-small    /* Fine print, captions */
--cs-font-size-small      /* Labels, metadata */
--cs-font-size-base       /* Body text (renamed from nothing) */
--cs-font-size-large      /* Subheadings */
--cs-font-size-x-large    /* Section headings */
--cs-font-size-2x-large   /* Page titles */

Option B: T-shirt abbreviated. Follows Tailwind/Chakra. Shorter, but abbreviations.

--cs-font-size-xs
--cs-font-size-sm
--cs-font-size-base
--cs-font-size-lg
--cs-font-size-xl
--cs-font-size-2xl

Given Corner Store's "spell things out" tendency everywhere else, Option A is more internally consistent. You lose the Tailwind familiarity of sm/lg but gain clarity for the maker audience. And x-small, small, large, x-large maps directly to CSS's own font-size keywords, which gives them prior art in the language itself.

6.6 Keep purpose-named spacing tokens (but understand the trade-off)

Corner Store's spacing tokens (--cs-spacing-container, --cs-spacing-section, --cs-spacing-listing) are purpose-named rather than scale-based. No other surveyed system does this. Every other system uses a generic spacing scale (--space-1, --spacing-md, --n-space-xl).

However, for ~45 tokens aimed at people who change values and don't compose new layouts, purpose-naming has a genuine advantage: a maker knows exactly what --cs-spacing-listing controls without CSS knowledge. The trade-off is that adding a new component requires adding new spacing tokens. At Corner Store's scale (~5 components, stable scope), this is manageable.

Recommendation: Keep purpose-named spacing for now. If the component count grows significantly, introduce a supplementary generic scale. But don't pre-optimize for a problem that doesn't exist yet.

6.8 Grammar summary for Corner Store

Based on the research, the recommended grammar for --cs- tokens is:

--cs-{category}-{concept}-{modifier}

Categories (what kind of property):
  surface, text, button, border, radius, shadow,
  font, spacing, transition, focus, layout

Concepts (what thing):
  page, listing, header, placeholder, section, container

Modifiers (which variant):
  hover, muted, tight, base, small, large, x-large

Not every token uses every level. The guiding principle: a maker who has never read the documentation should be able to guess what the token controls from its name alone.

Research Gaps and Methodology Notes

  • Nathan Curtis's canonical "Naming Tokens in Design Systems" article on Medium returned a 403 error when fetched. Claims attributed to Curtis come from search result excerpts and his other published writing. I could not verify exact quotes.
  • IBM Carbon's specific token rename mappings (v10 name -> v11 name, complete list) were referenced in migration docs but the GitHub markdown file hit a rate limit. The general pattern (numeric to descriptive) is confirmed across multiple sources.
  • Francesco Improta's "The Difference Between Background and Surface Tokens" article was behind a paywall. The distinction between these concepts is inferred from system implementations rather than this specific source.
  • The survey focuses on CSS custom property naming specifically. Systems like Chakra and Carbon also expose tokens through JavaScript APIs with different naming -- those APIs were not studied.
  • This report does not cover naming for animation, z-index, or breakpoint tokens, as those are outside Corner Store's current scope.
